# 03. 아키텍처 및 모듈 구조 계획 (G7 문서 정합 버전)

## 1) 핵심 준수 원칙 (module-basics / controllers / service-repository)

- 모듈 디렉토리명은 `vendor-module` 규칙을 따른다: `yjsoft-attendance`.
- 모듈은 `modules/_bundled/yjsoft-attendance` 기준으로 설계한다.
- 네임스페이스는 `Modules\Yjsoft\Attendance\`로 통일한다.
- 컨트롤러는 역할별 BaseController 상속 원칙을 지킨다.
  - 사용자 인증 API: `AuthBaseController`
  - 관리자 API: `AdminBaseController`
- Controller는 Service만 주입하고, Repository 직접 주입을 금지한다.
- Service는 Repository **Interface**를 주입받도록 설계한다(구현체 직접 타입힌트 금지).

## 3) 레이어 책임 고정

- Request(FormRequest): 입력 검증 전담 (컨트롤러/서비스 내 검증 로직 분산 금지)
- Controller: 인증/권한 경계 + Service 호출 + ResponseHelper 응답
- Service: 비즈니스 규칙(시간 제한, 연속 출석, 순위, 보너스)
- RepositoryInterface/Repository: 쿼리/영속성
- Model: 관계, cast, scope

## 4) 레이아웃/템플릿 구조 원칙 (module-layouts / layout-json)

- 모듈 레이아웃은 반드시 `resources/layouts/admin|user` 하위에 둔다.
- `layouts/` 루트 직배치는 스킵되므로 금지한다.
- 레이아웃 JSON 작성 시 HTML 태그 직접 사용 금지(기본 컴포넌트 사용).
- 레이아웃 액션 핸들러는 `apiCall`, `navigate`, `setState` 등 문서의 정식 핸들러만 사용한다.

## 5) 버전/릴리스 원칙

- 버전 변경 시 `CHANGELOG.md`를 Keep a Changelog 형식으로 동기 업데이트한다.
