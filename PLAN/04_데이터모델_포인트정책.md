# 04. 데이터 모델 및 포인트 정책

## 1) 테이블 설계 (초안)

### attendance_records

- id (PK)
- user_id (FK)
- attendance_date (DATE, 서버 기준 일자)
- checked_in_at (DATETIME)
- greeting_message (VARCHAR)
- day_rank (TINYINT, nullable: 1~3)
- base_points (INT)
- streak_bonus_points (INT)
- rank_bonus_points (INT)
- random_bonus_points (INT)
- total_points (INT)
- meta_json (JSON, 계산 근거)
- created_at / updated_at

제약:
- unique(user_id, attendance_date) : 1일 1회 강제
- index(attendance_date, checked_in_at) : 일일 순위 계산 최적화

### attendance_settings

- id (PK, 단일 row 또는 scope별 row)
- auto_checkin_enabled (BOOLEAN)
- checkin_time_start (TIME, nullable)
- checkin_time_end (TIME, nullable)
- random_bonus_enabled (BOOLEAN)
- random_bonus_probability (INT, 0~100)
- random_bonus_min_points (INT)
- random_bonus_max_points (INT)
- default_greetings_json (JSON)
- streak_week_bonus_points (INT)
- streak_month_bonus_points (INT)
- streak_year_bonus_points (INT)
- rank_1_bonus_points / rank_2_bonus_points / rank_3_bonus_points (INT)
- created_at / updated_at

## 2) 포인트 계산 순서

1. 기본 출석 포인트 산정
2. 연속 출석 보너스 산정
3. 일일 순위 보너스 산정
4. 랜덤 보너스 조건/확률 판정
5. 총합 계산 및 원자적 저장

## 3) 연속 출석 판정 로직

### 3-1. 일반 연속 출석(streak) 값

- streak 값은 UX 표시/기록용으로 유지한다.
- 전일 출석 시 streak +1, 아니면 1로 초기화한다.

### 3-2. 주간/월간/연간 보너스 판정 (핵심)

- 기간 보너스는 streak 길이로 판정하지 않는다.
- **달력 기간 완주**로 판정한다.
  - 주간: 해당 주의 모든 날짜 출석
  - 월간: 해당 월 1일~말일 전일 출석
  - 연간: 해당 연도 1/1~12/31 전일 출석
- 예시
  - 3/15~4/14 연속 출석: 월간 보너스 미지급
  - 3/1~3/31 출석: 월간 보너스 지급

### 3-3. 기간 보너스 중복 처리

- 동일 시점에 주간/월간/연간을 동시에 만족하면 1개만 지급한다.
- 우선순위는 **연간 > 월간 > 주간**.
- 저장 시 `meta_json`에 만족 후보/최종 선택 보너스를 기록한다.

## 4) 동시성/정합성

- 체크인 트랜잭션 내에서
  - 중복 출석 재검증
  - 순위 할당(select ... for update 또는 유사 잠금)
  - 기간 보너스 우선순위 단일화(연간 > 월간 > 주간)
  - 포인트 지급/로그 반영
- 실패 시 전부 롤백
